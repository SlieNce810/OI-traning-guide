# 08 - 贪心算法

> 贪心的本质不是"只看眼前"，而是"我能证明眼前最优不会错过全局最优"。

## 一、基础知识

### 1.1 核心定义

**贪心算法**：每步选择当前最优决策，期望最终得到全局最优。关键约束：必须能**证明**局部最优 → 全局最优。若无法证明，结果可能完全错误。

### 1.2 工作原理

典型流程：排序（贪心最常见的预处理）→ 按标准依次选中 → 累计结果。

**反例**：硬币面额 [1,3,4]，凑 6 分：贪心 4+1+1=3 枚，最优 3+3=2 枚。贪心失败！

### 1.3 适用场景

| 场景 | 前提 |
|------|------|
| 区间调度 | 按结束时间排序，权重相同 |
| 霍夫曼编码 | 最优前缀码的数学保证 |
| 最小生成树 | 图论定理保证 |
| 跳跃游戏 | 每步收益可累加 |

**核心前提**：问题具有**贪心选择性质**（局部最优能构成全局最优）和**最优子结构**。

> 💡 **最优子结构（Optimal Substructure）详解**：
> 一个问题有最优子结构，意味着：**最优解包含了子问题的最优解**。
> 
> 例如活动选择问题中，如果最优解包含了活动 A，那么"在 A 结束后剩余时间里的最优安排"也是那个子问题的最优解。如果它不成立（子问题的最优解组合起来不是全局最优），就不能用贪心或 DP。
>
> 🔑 **反例续**：硬币 [1,3,4] 凑 6 分，贪心选 4 后还需凑 2 分（用两张 1，共 3 枚），但最优是 3+3=2 枚。贪心失败的原因是"选 4 的剩余子问题"的最优解（2 枚 1）不是全局最优解的一部分——最优子结构存在但贪心选择性质不成立。

### 1.4 优缺分析

**优点**：O(n log n)（排序），代码简洁
**缺点**：必须证明正确性，否则是错误算法；约束稍变可能失效

## 二、贪心 vs DP 的本质区别

| | 贪心 | DP |
|--|------|-----|
| 决策方式 | 每步选当前最优 | 考虑所有可能 |
| 回溯 | 不回头 | 比较所有方案 |
| 正确性 | 需要证明 | 总是正确（前提状态定义对） |
| 效率 | 更快 | 可能较慢 |

**经验**：不确定贪心是否对 → 用 DP。DP 对了再考虑能否贪心优化。

## 三、如何证明贪心正确性

贪心最难的不是写代码，而是**证明正确**。两种标准证明方法：

### 方法一：交换论证法

假设最优解不同于贪心解，通过"交换"两元素证明贪心解不劣于最优解。

**例——活动选择问题**：贪心选最早结束的活动。假设最优解选了活动 A 而非贪心选的 B（B 结束更早）。将 A 换成 B，B 结束更早 → 不会减少后续可选活动数 → 新解不差于原最优解。归纳可得贪心最优。

### 方法二：归纳法

证明：如果前 k 步贪心选择是最优解的一部分，则第 k+1 步也是。

## 四、常见贪心模式

### 模式 1：排序 + 扫描

**LeetCode 435. 无重叠区间**：删除最少区间使不重叠 → 按结束时间排序，贪心选最早结束的。

```rust
fn erase_overlap_intervals(mut intervals: Vec<Vec<i32>>) -> i32 {
    // 按结束时间排序：结束越早，留给后面的空间越多
    intervals.sort_unstable_by_key(|v| v[1]);
    let mut end = intervals[0][1];
    let mut removed = 0;

    for i in 1..intervals.len() {
        // 当前区间起点 < 上一个选中区间的终点 → 重叠，需要删除
        if intervals[i][0] < end {
            removed += 1;
        } else {
            end = intervals[i][1]; // 不重叠，更新终点
        }
    }
    removed
}
```

**为什么按结束时间排序是对的**：选最早结束的活动，给后面留最多剩余时间段——这是"不要挡路"的贪心哲学。

### 模式 2：两次遍历（左右各一次）

**LeetCode 135. 分发糖果**：每人至少 1 颗，评分高的比相邻多。

```rust
fn candy(ratings: Vec<i32>) -> i32 {
    let n = ratings.len();
    let mut candies = vec![1i32; n]; // 每人至少 1 颗

    // 左→右：确保评分比左边高的孩子糖果更多
    for i in 1..n {
        if ratings[i] > ratings[i-1] {
            candies[i] = candies[i-1] + 1;
        }
    }
    // 右→左：确保评分比右边高的孩子糖果更多
    for i in (0..n-1).rev() {
        if ratings[i] > ratings[i+1] {
            candies[i] = candies[i].max(candies[i+1] + 1);
        }
    }
    candies.iter().sum()
}
```

**设计原理**：相邻关系是双向的。一次遍历只能满足一个方向，两次遍历取最大值同时满足左右。

### 模式 3：维护最远可达

**LeetCode 55. 跳跃游戏**：从索引 0 出发，nums[i] 表示可跳最大距离，判断能否到末尾。

```rust
fn can_jump(nums: Vec<i32>) -> bool {
    // farthest: 当前能到达的最远位置
    let mut farthest = 0usize;
    for i in 0..nums.len() {
        // 当前位置不可达 → 后面更不可达
        if i > farthest { return false; }
        farthest = farthest.max(i + nums[i] as usize);
    }
    true
}
```

**核心洞察**：我们关心的不是"跳到哪里"，而是"最远能到哪里"——这就是贪心的选择指标。

## 五、常见错误

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 不证明就用贪心 | WA（错误答案） | 用 DP 验证或用数学方法证明 |
| 贪心指标选错 | 全盘错误 | 找反例测试自己的贪心策略 |
| 排序关键字选错 | 结果不对 | 仔细分析"什么最早结束""什么边际收益最大" |
| 和 DP 混淆 | 舍最优选用次优 | 不确定时先写 DP |

## 六、面试识别速查

| 关键词 | 算法 |
|--------|------|
| "最少/最多" + "排序后扫描" | 贪心 |
| "区间" + "不重叠" | 排序 + 贪心 |
| "跳跃" + "可达" | 贪心维护最远可达 |
| "糖果/分配" + "相邻约束" | 两次遍历贪心 |
| 面试中如果不确定 → | 先用 DP，再尝试贪心 |
